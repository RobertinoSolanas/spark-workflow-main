# Plan: Deploy Spark Workflow on MicroK8s with ArgoCD using Helm

## Goal
Deploy the entire Spark Workflow stack on MicroK8s using Podman for image building, the MicroK8s local image registry, Helm for Kubernetes descriptors, and ArgoCD for GitOps-driven deployment — all within the `spark-workflow` namespace.

---

## Current State Analysis

### What Exists
- **3 service compose files**: `docker-compose.yaml` (infrastructure) and `docker-compose.services.yaml` (application services)
- **11 custom-built images** defined across:
  - `02-backend/` — 7 backend services (agent_orchestration_service, document_management_service, formal_completeness_check, plausibility_notes, project_logic_service, temporal_codec_service, comment_service)
  - `05-modulcluster/` — 3 AI modules (modul-inhaltsextraktion, modul-formale-pruefung, modul-plausibilitaet-pruefung)
  - `04-shared-services/basiskomponenten/` — 2 shared services (litellm-proxy, unoserver)
- **Pre-built images** (no custom Dockerfile): temporal, elasticsearch, postgres, minio, qdrant, tempo, loki, prometheus, grafana, otel-collector, temporal-admin-tools, temporal-init, minio-init, docling-serve, plaus
- **Existing scripts**: `scripts/build-images.sh` and `scripts/deploy-argocd.sh` (already designed for podman → microk8s)
- **No Helm charts exist** — deployment folder `deployment/` and `deployment/argocd/` are empty (only README.md files, both empty)
- No k8s manifests, no Helm charts, no existing Helm structure

### What Is Missing
1. **MicroK8s local registry** — not configured yet
2. **Helm charts** — need to be created from scratch for both `deployment/` and referenced in `deployment/argocd/`
3. **ArgoCD HelmApplication/App manifests** — in `deployment/argocd/`
4. **Build/import/push pipeline** — needs to push to MicroK8s local registry instead of using `ctr images import`
5. **Helm chart structure** — complete `Chart.yaml` and `values.yaml` for each layer

### Step 1: Enable MicroK8s Local Registry

```bash
# Enable the microk8s local registry addon
microk8s enable registry

# Verify registry is running
microk8s kubectl get pods -n microk8s-registry
```

The MicroK8s local registry runs at `localhost:32000` (or the internal registry service endpoint). Images are tagged as `localhost:32000/<image-name>:<tag>`.

### Step 4: Push Images to MicroK8s Local Registry

```bash
# Tag and push each image to the local registry
podman tag postgres:16.10 localhost:32000/spark-infra-postgresql:16.10
podman push localhost:32000/spark-infra-postgresql:16.10

# Same pattern for all images (already built in Step 3)
```

For the MicroK8s local registry, images pushed to `localhost:32000/` are automatically available to pods in the cluster with no `imagePullPolicy` changes needed (the registry is a cluster-internal service).

### Step 6: Create Single ArgoCD Application

Create one ArgoCD Application in `deployment/argocd/spark-workflow.yaml` that references the **single parent Helm chart** (which internally manages all 4 sub-charts):

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: spark-workflow
  namespace: argocd
  labels:
    app.kubernetes.io/part-of: spark-workflow
spec:
  project: default
  source:
    repoURL: <git-repo-url>
    path: deployment/charts/spark-workflow
    targetRevision: main
    helm:
      valueFiles:
        - values.yaml
  destination:
    server: https://kubernetes.default.svc
    namespace: spark-workflow
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
      - CreateNamespace=true
```

All sub-charts (infra, backend, ai-modules, shared) are managed through the **parent chart's** single ArgoCD Application — ArgoCD handles the dependency ordering automatically.

### Step 7: Create Deployment Script

Create `deployment/deploy.sh` that:

1. Checks prerequisites (podman, microk8s, local registry, argocd)
2. Runs `scripts/build-images.sh` to build all images
3. Tags all images for the MicroK8s local registry (localhost:32000)
4. Pushes all images to `localhost:32000`
5. Runs `helm template` to dry-run and validate
6. Runs `helm upgrade --install` for each chart layer
7. Creates ArgoCD Application resources from `deployment/argocd/`
8. Verifies all pods are ready in `spark-workflow` namespace
9. Reports access URLs for all services

## Key Technical Decisions

1. **Use MicroK8s local registry (localhost:32000)** instead of `imagePullPolicy: Never` + podman save/ctr import. The registry add-on provides a proper cluster-internal registry that Kubernetes services can pull from with standard `imagePullPolicy: Always`. This matches the request to "check local registry of microk8s" and use it for all pushes.

2. **Helm charts** — All Kubernetes descriptors use Helm (as requested). A parent-child chart hierarchy is used for clean separation of concerns:
   - Parent chart `spark-workflow` orchestrates all sub-charts
   - Sub-charts: `spark-infra`, `spark-backend`, `spark-ai-modules`, `spark-shared`

3. **Namespace** — Everything uses `spark-workflow` namespace as specified.

4. **ArgoCD** — Uses `helm` source type in ArgoCD Application manifests pointing to the `deployment/charts/` path. The `app-of-apps` pattern enables managing all application deployments as a single unit.

5. **imagePullPolicy: Always** — Since we're using the MicroK8s local registry, images are pulled by policy (not `Never`). After pushing a new image to localhost:32000, ArgoCD's self-heal or manual sync picks up the change.
